# cc-haha 安全加固记录

- **项目路径**：`/Users/karekin/Downloads/coding/project/cc-haha`
- **记录时间**：2026-04-04
- **操作目标**：在**尽量保留完整功能与高权限能力**的前提下，对项目进行低侵入式安全加固，降低已知依赖漏洞与供应链风险。

---

## 一、加固前的主要风险判断

本次加固前，项目主要存在以下风险：

1. **已知 Node 依赖漏洞**
   - `@anthropic-ai/sdk@0.80.0`
   - `lodash-es@4.17.23`

2. **Node 依赖供应链来源不理想**
   - 锁文件大量引用 `registry.npmmirror.com`
   - 不利于生产环境的统一供应链审计与信任收敛

3. **Computer Use Python 运行时存在动态安装链路**
   - `runtime/requirements.txt` 使用 `>=` 版本范围
   - `src/utils/computerUse/pythonBridge.ts` 在运行时自动创建 venv 并执行 `pip install`
   - 原实现还包含 `pip install --upgrade pip`，会额外引入一层远程升级面

4. **项目本身仍属于高权限代理型 CLI**
   - 支持 Bash 执行、插件/marketplace、自动更新、Computer Use 等能力
   - 因此本次加固的目标是**降低风险，不是把它改造成低权限安全沙盒**

---

## 二、本次加固原则

本次操作遵循以下原则：

- **不阉割完整功能**
- **不关闭高权限能力**
- **优先修复已知漏洞与供应链问题**
- **优先做低侵入、可回滚、可验证的变更**

因此，本次没有禁用以下能力：

- `--dangerously-skip-permissions`
- marketplace / plugin 能力
- Computer Use
- 完整 CLI / TUI 功能

## 三、实际执行的加固操作

### 1）升级存在漏洞的直接依赖

修改文件：`package.json`

执行内容：

- 将 `@anthropic-ai/sdk` 从 `^0.80.0` 升级到 `^0.82.0`
- 将 `lodash-es` 从 `^4.17.23` 升级到 `^4.18.1`

变更目的：

- 消除 `npm audit` 报出的已知漏洞
- 降低已知路径校验问题与 lodash 相关高危风险

---

### 2）固定仓库默认 npm 源为官方 registry

新增文件：`.npmrc`

内容：

```ini
registry=https://registry.npmjs.org/
```

变更目的：

- 避免继续使用第三方镜像源作为默认依赖来源
- 让后续安装、审计、锁文件生成统一走官方 npm registry

---

### 3）重建 Node 锁文件

操作对象：

- `package-lock.json`
- `bun.lock`

执行内容：

- 使用官方 registry 重新安装/生成锁文件
- 清除原锁文件中 `registry.npmmirror.com` 的引用

加固效果：

- `package-lock.json` 中 `registry.npmmirror.com` 命中数已降为 **0**
- `bun.lock` 中 `registry.npmmirror.com` 命中数已降为 **0**

---

### 4）将 Python Computer Use 依赖改为精确版本 pin

修改文件：`runtime/requirements.txt`

原来：

```txt
mss>=10.1.0
Pillow>=11.3.0
pyautogui>=0.9.54
pyobjc-core>=11.1
pyobjc-framework-Cocoa>=11.1
pyobjc-framework-Quartz>=11.1
```

修改后：

```txt
mss==10.1.0
Pillow==11.3.0
pyautogui==0.9.54
pyobjc-core==11.1
pyobjc-framework-Cocoa==11.1
pyobjc-framework-Quartz==11.1
```

变更目的：

- 避免运行时因版本范围解析导致非预期升级
- 提高运行结果可复现性
- 降低 Python 侧供应链漂移风险

补充说明：

本次选定的版本均可在当前可查询结果中确认存在。

---

### 5）加固 Python bridge 的运行时安装逻辑

修改文件：`src/utils/computerUse/pythonBridge.ts`

#### 5.1 新增 requirements pin 校验

新增逻辑：

- 默认要求 `runtime/requirements.txt` 中依赖使用 `==` 精确 pin
- 若存在未 pin 的依赖，则启动时报错，拒绝继续安装
- 仅在显式设置 `CLAUDE_CODE_ALLOW_UNPINNED_RUNTIME_REQUIREMENTS=1` 时放开

目的：

- 防止未来维护中把 requirements 再改回宽松范围却未被发现

#### 5.2 pip 安装时加入更保守参数

安装参数调整为：

- `--require-virtualenv`
- `--disable-pip-version-check`
- `--no-input`

同时增加 pip 环境变量：

- `PIP_DISABLE_PIP_VERSION_CHECK=1`
- `PIP_NO_INPUT=1`
- `PYTHONDONTWRITEBYTECODE=1`
- `PYTHONNOUSERSITE=1`

目的：

- 避免交互式安装
- 避免使用用户站点包污染运行环境
- 减少无关网络检查
- 保持依赖安装发生在项目私有 venv 内

#### 5.3 删除自动升级 pip 的步骤

原实现会先执行：

```bash
python -m pip install --upgrade pip
```

本次已移除。

目的：

- 减少额外的远程升级供应链面
- 避免 bootstrap 过程中引入非必要的最新版 pip 变量
- 保持安装链路更短、更稳定

## 六、加固后仍保留的风险

尽管本次已完成基础加固，但以下风险**仍然存在**，因为它们属于产品能力边界本身，而不是简单依赖问题：

1. **高权限 Bash 执行能力仍然存在**
2. **插件 / marketplace / 远程拉取能力仍然存在**
3. **自动更新能力本身仍然可用（如果配置开启）**
4. **Computer Use 仍然可以控制桌面环境**
5. **完整功能模式下，项目本质仍然是高权限代理 CLI，而不是低权限线上服务**

因此：

- **适合**：自用、本地、隔离容器、受控内网环境
- **不适合**：未经进一步裁剪就直接公网暴露为共享服务

## 七、建议的后续动作

如果后续还要继续提升安全性，可按优先级继续做：

1. 为生产部署单独做一套“受限配置”
2. 禁用不必要的远程 marketplace 与插件来源
3. 对 Python requirements 增加 hash pin（`--require-hashes` 级别）
4. 将 Node / Python 依赖安装切换到私有制品源
5. 对外联域名做白名单限制
6. 将高权限模式仅保留给本地自用，生产环境采用受限权限运行
